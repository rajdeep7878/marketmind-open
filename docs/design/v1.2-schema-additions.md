# v1.2 — schema additions surfaced by the 9-hunt era

**Status:** DESIGN PASS (2026-05-24). Locked branch: `v1.2-schema-additions`.
Implementation deferred to a separate session.

**TL;DR:** The 9-hunt strategy-discovery era surfaced **3 hard schema gaps**
and **1 soft gap** at the extraction layer (plus 4 gauntlet rejections at
the strategy-quality layer — separate concern). The design pass discovered
that **2 of the 6 originally-scoped primitives are already shipped** in
v2 schema, reducing v1.2's real scope to **4 new primitives**, sub-phased
across **6 sub-phases (v1.2.A → v1.2.F)** with an estimated **~18 commits**
on a `v1.2-schema-additions` branch. Container rebuild discipline (api +
worker + trader_worker) applies post-merge, per Phase B's hard-won
learning.

## §1 Context — empirical demand signal

The hunt-era ran 9 real-strategy extractions (Phase A's Modern Turtle was
the only one to seed; B.7, B.9, and seven post-Phase-B hunts were all
rejected at one of two layers):

| layer | rejections | reason shape |
|---|---|---|
| gauntlet (strategy-quality) | 4 | over-permissive entries → catastrophic capital loss on real BTC data |
| extraction (schema-coverage) | 3 + 1 soft | LLM honestly flagged the schema can't express a load-bearing primitive |

Of the 3 hard extraction-layer rejections, two surfaced as
`partially_extractable` with the LLM noting a "lossy approximation" of the
real primitive (Hunt 3's percentile, Hunt 5's bars-since-last-trade), and
one as `not_extractable` after the constructed spec failed strict
validation (Hunt 6B's hour-of-day session filter).

**Rate:** 3 hard gaps / 9 extractions = **33 % schema-gap discovery rate**.
That's high enough that adding the missing primitives unblocks a real
class of strategies (intraday seasonality, percentile-band regimes,
time-based throttles); it's also low enough that the schema's overall
coverage is good (the gaps are real but bounded).

The "let real-strategy extraction surface schema gaps" pattern is the same
discipline as Phase A's "extraction prompt teaching audits" finding —
real-world inputs are the most efficient way to find what's missing. The
v1.2 batch is the response to that signal, deliberately ordered by
empirical demand.

## §2 Goals and non-goals

### Goals

- Add 4 new schema primitives (`percentile()` expression, `bars_since_last`
  prior_trade predicate, `TimeOfDayCondition`, `DayOfWeekCondition`,
  `TakeProfitAtrMultiple`) to v2.
- Preserve 100 % backward compatibility — v1 strategies bit-identical;
  Phase A state primitives unchanged; Phase B FeeModel + SlippageModel +
  sqrt(N) drift scaling preserved.
- Each primitive ships with: schema + iterative engine integration + vbt
  translator integration (where applicable) + extraction prompt teaching
  + drift parity test + per-primitive unit tests.
- Total test count grows from 1099 → 1130+ (estimate +30 new tests, no
  removals).
- Container rebuild discipline documented for post-merge deployment (the
  Phase B finding: api + worker + trader_worker all need rebuild after
  any schema field addition).

### Non-goals

- Strategy hunting — v1.2 is schema work, not gauntlet-passing-strategy
  work. Hunting resumes after v1.2 ships.
- Tier-2 / Tier-3 / Tier-4 primitives listed in the brief — defer to
  v1.3+ unless they surface in real hunts.
- `spot` exchange-string extraction quirk — documented separately in
  `docs/operations/v1.1-todos.md`; not part of v1.2 schema scope (it's a
  prompt-teaching audit + a `_exchange_key()` mapping update).
- `log_returns` indicator variant (Hunt 6C soft gap) — at 1H crypto
  density `r_simple ≈ r_log` for typical per-bar returns (<2 %); the
  approximation is non-material. Defer to v1.3 if quant-paper extractions
  remain common and the gap surfaces again.
- Architectural changes — every primitive in v1.2 is bounded-effort
  schema + integration + tests. No new engines, no new sibling stepper,
  no new persistence tables.

### v1.2.A — `percentile()` indicator/expression

**Surfaced by:** Hunt 3 (Momentum + ATR-percentile, 2026-05-24).

**Pattern:** rolling empirical percentile of an indicator series at the
current bar. The Hunt 3 strategy needed "ATR(14) is between the 30th
and 70th percentile of ATR(14) values over the last 168 hours" — a
band-defined regime in distributional terms, not in fixed-threshold or
ratio terms.

**Shape (proposed):** a new `ExpressionKind` (NOT a new `IndicatorName` —
percentile is a transformation over an existing expression, not a
standalone indicator). Mirrors how `lagged` and `scaled` are wrapper
expressions in the existing `expressions.py`:

```python
class PercentileExpr(_StrictModel):
    """The rolling empirical percentile (0..1) of an expression over a
    trailing window, at the current bar. Useful for regime detection
    where the threshold should be expressed in distributional terms
    rather than fixed numbers.
    """
    kind: Literal["percentile"] = "percentile"
    expression: Expression  # the inner expression to percentile
    window: int = Field(ge=10, le=10_000)  # trailing bar count
```

**Schema file:** `shared/src/marketmind_shared/schemas/strategy_spec/expressions.py`
gains the `PercentileExpr` class and adds it to the `Expression` union.

**Iterative engine integration:** `workers/src/marketmind_workers/backtest/`
`indicators.py` evaluators add one helper:

```python
def percentile_rolling(series: pd.Series, window: int) -> pd.Series:
    # pd.Series.rolling(window, min_periods=window).rank(pct=True) returns
    # the rank-as-fraction of the most recent value within the window.
    return series.rolling(window=window, min_periods=window).rank(pct=True)
```

The expression evaluator dispatch in
`workers/src/marketmind_workers/backtest/translator.py` adds one branch
under `_evaluate_expression(expr, ctx)` to call the helper. One-line
addition pattern, same as `lagged` and `scaled`.

**vbt translator integration:** the vbt path consumes pre-computed
indicator series; the percentile transformation is computed once at the
translator level and feeds into the existing comparison machinery. No
special vbt code needed — it's just another pre-computed series.

**Extraction prompt teaching:** new section under "## SCHEMA REFERENCES"
in `workers/src/marketmind_workers/services/extraction_prompt.py` titled
"### percentile — rolling empirical percentile of an expression". Two
worked examples (the load-bearing pattern from Phase A's hard-won
knowledge):

1. **ATR-percentile regime** (Hunt 3's failed case):
   ```json
   {"kind": "percentile",
    "expression": {"kind": "indicator", "name": "atr",
                   "params": {"period": 14}},
    "window": 168}
   ```
   Then compare to `0.7` / `0.3` for the percentile-band regime.

2. **RSI-percentile** (rank-based RSI instead of absolute):
   ```json
   {"kind": "percentile",
    "expression": {"kind": "indicator", "name": "rsi",
                   "params": {"period": 14}},
    "window": 720}
   ```

Plus the standard warning: percentile of a stateful indicator (e.g.
Supertrend.direction) is degenerate — the inner expression should be
continuous-numeric.

**Drift parity:** the existing four drift parity gates (Supertrend
Tier-2, Turtle prior_signal Tier-3, Turtle prior_trade Tier-3, the
sibling-evaluator pattern) all use specs that don't include
`percentile`. Adding the primitive doesn't touch any existing code
path — bit-identity preserved by construction. **New gate** (this
sub-phase): a 3-bar test fixture exercising percentile() in both the
vbt translator and the iterative engine, asserting they produce
identical numeric output. The percentile primitive doesn't introduce
state (it's a pure rolling window), so a sibling-evaluator drift gate
isn't needed; the vbt-vs-iterative output identity is sufficient.

**Test plan:**
- `test_expression_percentile_basic.py` — percentile of a known series
  produces the expected rank-fraction (e.g., percentile of [1,2,3,4,5]
  at window=5 returns [NaN,NaN,NaN,NaN,1.0]; at window=3 returns
  [NaN,NaN,1.0,1.0,1.0]).
- `test_expression_percentile_edge_cases.py` — window > series length
  produces NaN; constant input produces 1.0 across the window;
  near-zero values; large windows.
- `test_extraction_percentile.py` — extract the Hunt 3 strategy
  (Momentum + ATR-percentile) and verify the spec now contains a
  `PercentileExpr` instead of falling back to `highest()` proxy.
- `test_translator_percentile_in_compare.py` — a spec with a percentile
  comparison validates, builds signals, and runs through the engine.
- `test_iterative_percentile_drift_parity.py` — vbt path and iterative
  path produce bit-identical signals + trades for a no-state percentile
  spec.

**Commit count:** 4 (schema + helper + translator/engine + tests +
prompt teaching = naturally 4 commits with the per-commit-tests
pattern). Plus 1 commit for `_detect_axes` widening to include `window`
parameter for percentile (overfitting parameter sweep coverage).
Total: **5**.

**Risk:** low. Pure functional addition; no state, no migration, no
trader-side change beyond the schema visibility.

### v1.2.B — `prior_trade(bars_since=N)` predicate extension

**Surfaced by:** Hunt 5 (Mean-rev + Tier-3 throttle, 2026-05-23).

**Pattern:** time-based re-entry throttle, distinct from the existing
outcome-based predicates (`last_won` / `last_lost` /
`consecutive_wins_at_least` / `consecutive_losses_at_least`). A mean-
reversion strategy needs to "wait at least N bars after the last
completed trade before considering a new entry" — common discipline
that catches whipsaw re-entries.

**Shape (proposed):** extend the `PriorTradeCondition.predicate` enum
with one new literal and add a corresponding branch in
`TradeHistory.evaluate_predicate()`. The `n` field already exists on
the schema; its semantic for the new predicate is "at least N bars
elapsed since the last completed trade's exit bar".

`shared/src/marketmind_shared/schemas/strategy_spec/conditions.py`:

```python
predicate: Literal[
    "last_won",
    "last_lost",
    "consecutive_losses_at_least",
    "consecutive_wins_at_least",
    "bars_since_last_at_least",  # NEW — time-based throttle
] = Field(...)
```

The `n` field's existing `description` ("Run length for the consecutive_*
predicates; ignored by last_won / last_lost") gets a sentence appended:
"For `bars_since_last_at_least`, n is the minimum bar count since the
last trade's exit; the predicate is true when at least n bars have
elapsed."

**Iterative engine integration:** the predicate evaluator at
`workers/src/marketmind_workers/backtest/trade_history.py` —
`TradeHistory.evaluate_predicate(predicate, n)` — gains one branch.
The `TradeRecord` shape already records `exit_bar`; the new branch
just compares `current_bar - last_exit_bar >= n`:

```python
if predicate == "bars_since_last_at_least":
    last = self.last_trade()
    if last is None:
        return False  # no prior trade — predicate is False (same convention as last_won)
    return (current_bar - last.exit_bar) >= n
```

Note: the existing `evaluate_predicate(predicate, n)` signature doesn't
take `current_bar`. This is a real signature change — **needs careful
threading** through the call sites in the iterative engine and the
live stepper. Likely 3-4 call sites; small refactor.

**vbt translator integration:** none — `prior_trade` is already T3-only
(not supported in vbt). The new predicate is a member of the same
class; the router (`spec_uses_tier3`) already gates the spec to the
iterative engine.

**Extraction prompt teaching:** the prompt's "### prior_trade — outcome-
based gates on prior trade results" section gets one more worked
example:

```
A 24-bar re-entry throttle — wait at least 24 bars after the last trade's
exit before considering a new entry. Common discipline for mean-reversion
strategies; prevents whipsaw re-entries.

  {"type": "prior_trade", "predicate": "bars_since_last_at_least", "n": 24}

Combine with NOT() to express "only enter if at least 24 bars have
elapsed since the last trade".
```

**Drift parity:** the existing Turtle prior_signal + prior_trade gates
use specs without `bars_since_last_at_least`. Adding the predicate
doesn't touch their evaluation. **New gate** (this sub-phase): a
specifically-crafted Tier-3 spec using `bars_since_last_at_least` runs
through both the iterative one-shot path and the iterative_live sibling
stepper with `Tier3State` JSON-round-tripped per cycle, asserting trade
ledger bit-identity.

**Test plan:**
- `test_trade_history_bars_since_last.py` — pure unit tests against
  `TradeHistory.evaluate_predicate("bars_since_last_at_least", n)`:
  - No prior trade → False
  - 1 prior trade closed at bar 100, current bar 110, n=24 → False
  - 1 prior trade closed at bar 100, current bar 124, n=24 → True
  - 1 prior trade closed at bar 100, current bar 125, n=24 → True
- `test_extraction_bars_since_last.py` — extract Hunt 5's strategy and
  verify the spec now contains `bars_since_last_at_least` instead of
  the `consecutive_wins_at_least` proxy.
- `test_iterative_bars_since_throttle.py` — small fixture spec that
  exercises the throttle: confirm trades are spaced ≥ n bars apart.
- `test_iterative_live_drift_parity_bars_since.py` — new drift parity
  gate (mirrors `test_iterative_live_drift_parity_prior_trade.py`).

**Commit count:** 3 (schema + impl + tests). The drift parity gate
typically ships as a 4th commit per Phase A's pattern. Total: **4**.

**Risk:** low–medium. The `evaluate_predicate` signature change needs
careful threading; `current_bar` must be available at every call site
(it is — the iterative engine tracks it explicitly). The schema
addition is enum-only, no migration.

### v1.2.C — `TimeOfDayCondition`

**Surfaced by:** Hunt 6B (intraday seasonality, 2026-05-24) — `not_extractable`
because the constructed spec failed strict validation (no
hour-of-day variant in the `Condition` union).

**Pattern:** boolean condition that is True only when the current bar's
`open_ts` falls within a configured hour range (UTC). Used for
intraday-seasonality strategies, session filters, and end-of-day
flatten rules.

**Shape (proposed):** new `Condition` variant in the discriminated
union:

```python
class TimeOfDayCondition(_StrictModel):
    """True only when the current bar's open_ts falls within the
    configured UTC hour range. Inclusive of `start_hour_utc`, exclusive
    of `end_hour_utc`. Supports wrap-around windows (start > end means
    "from start through midnight to end the next day").
    """
    type: Literal["time_of_day"] = "time_of_day"
    start_hour_utc: int = Field(ge=0, le=23)
    end_hour_utc: int = Field(ge=0, le=23)
    # Note: equal start and end => empty window (always False);
    # validator emits a warning if start == end.
```

**Schema file:** `shared/src/marketmind_shared/schemas/strategy_spec/conditions.py`
gains the class and adds it to the `Condition` union.

**Iterative engine integration:** `workers/src/marketmind_workers/backtest/`
`translator.py` — the condition evaluator dispatch (`_compile_condition`)
adds one branch:

```python
if isinstance(cond, TimeOfDayCondition):
    hours = ctx.primary_df.index.hour  # tz-aware UTC DatetimeIndex
    if cond.start_hour_utc <= cond.end_hour_utc:
        return pd.Series((hours >= cond.start_hour_utc) & (hours < cond.end_hour_utc),
                          index=ctx.primary_df.index)
    else:
        # Wrap-around: e.g. start=22, end=2 → 22, 23, 0, 1
        return pd.Series((hours >= cond.start_hour_utc) | (hours < cond.end_hour_utc),
                          index=ctx.primary_df.index)
```

The vbt path consumes a pre-computed boolean series — no special vbt
code needed.

**Iterative path:** uses the same pre-computed series via the existing
`_build_entry_evaluator` machinery; no additional code.

**Extraction prompt teaching:** new section under "## SCHEMA REFERENCES"
titled "### Time-of-day and day-of-week — intraday and weekly gates".
Worked examples:

1. **Intraday seasonality** (Hunt 6B's failed case):
   ```json
   {"type": "time_of_day", "start_hour_utc": 22, "end_hour_utc": 24}
   ```
   "Hold long only during the 22:00 and 23:00 UTC hourly bars
   (end_hour_utc is exclusive, so 22-24 means 22:00 and 23:00)."

2. **End-of-day flatten** (a common discipline):
   ```json
   {"type": "not",
    "condition": {"type": "time_of_day",
                  "start_hour_utc": 23, "end_hour_utc": 24}}
   ```
   "Exit any open position by 23:00 UTC — useful for risk-off-overnight
   strategies."

**Drift parity:** the time-of-day primitive is stateless (depends only
on the bar's index timestamp); no sibling-evaluator concern. **New
gate** (this sub-phase): a fixture spec using `time_of_day` runs
through vbt and iterative paths producing identical results.

**Trader-side concern:** the trader's signal_engine ingests
trader_candles with tz-aware UTC `open_ts`; the same evaluator
machinery applies. **No special trader integration**, but a smoke test
in `test_trader_spec_template.py` for a time-of-day spec is worth
adding to confirm the live cycle handles it.

**Test plan:**
- `test_condition_time_of_day_basic.py` — given a 1H DataFrame indexed
  by UTC hours, the predicate returns True for the configured hours and
  False otherwise. Include the wrap-around case (start=22, end=2 means
  22, 23, 0, 1 all True).
- `test_extraction_time_of_day.py` — extract Hunt 6B's strategy and
  verify the spec now contains `time_of_day` and validates cleanly.
- `test_translator_time_of_day_runs_e2e.py` — minimal time-of-day spec
  produces signals at the right bars.
- `test_iterative_time_of_day.py` — same spec runs through iterative
  path; signal bars match vbt path.
- `test_trader_spec_template_time_of_day.py` — synthetic version with
  a time-of-day condition loads + evaluates cleanly through the
  signal_engine (mirrors B.5's synthetic-version pattern, but exercises
  the new condition specifically).

**Commit count:** 3 (schema + integration + tests + prompt teaching as
one combined doc commit). Total: **3**.

**Risk:** low. Stateless primitive; the timestamp index is already
tz-aware UTC throughout the codebase; the wrap-around case is the only
edge worth a dedicated test. The trader-side smoke test is the
load-bearing add — without it, a future time-of-day strategy seeded
in production could surface a tz-handling bug at runtime.

### v1.2.D — `DayOfWeekCondition`

**Surfaced by:** not directly, but same primitive family as v1.2.C and
near-zero marginal cost given v1.2.C's implementation.

**Pattern:** boolean condition that is True only when the current bar's
`open_ts` falls on a configured set of weekdays (Mon=0..Sun=6, ISO
calendar UTC). Used for weekend-effect, Monday-effect, day-of-week
seasonality patterns.

**Shape (proposed):**

```python
class DayOfWeekCondition(_StrictModel):
    """True only when the current bar's open_ts (UTC) falls on one of
    the configured weekdays. ISO weekday convention: Monday=0..Sunday=6.
    """
    type: Literal["day_of_week"] = "day_of_week"
    # Set of allowed weekdays, 0..6. At least one element required.
    weekdays: list[int] = Field(min_length=1, max_length=7)
    # Validator: weekdays must be unique and all in [0, 6].
```

**Implementation symmetric to v1.2.C.** Iterative engine evaluator:

```python
if isinstance(cond, DayOfWeekCondition):
    weekdays = ctx.primary_df.index.weekday  # 0..6
    return pd.Series(weekdays.isin(cond.weekdays), index=ctx.primary_df.index)
```

**Extraction prompt teaching:** combined with v1.2.C in the same
"### Time-of-day and day-of-week" section. One worked example:

```json
{"type": "day_of_week", "weekdays": [0, 1, 2, 3, 4]}
```
"Trade only on weekdays (skip Sat / Sun). Crypto trades 24/7 but
exchange-deposit / institutional activity drops on weekends, so
weekend-only-strategies and weekday-only-strategies both exist in
practice."

**Drift parity + trader-side:** identical to v1.2.C. Combined test
file.

**Test plan:**
- `test_condition_day_of_week_basic.py` — predicate returns True for
  configured weekdays, False otherwise. Validator rejects empty,
  duplicate, out-of-range weekdays.
- `test_iterative_day_of_week.py` — minimal spec with a Monday-only
  condition; signals only fire on Mondays.
- (no separate extraction test — combined with v1.2.C's; the LLM
  extraction surface is the same: "day of week" gates.)

**Commit count:** 2 (schema + impl + tests in a tight commit; prompt
update is part of v1.2.C's doc commit). Total: **2**.

**Risk:** minimal. Symmetric to v1.2.C.

### v1.2.E — `TakeProfitAtrMultiple` exit (only the ATR half is new)

**Surfaced by:** not directly in the hunt era, but a standard quant
exit primitive symmetric to the existing `StopLossAtrMultiple`. The
percent (`TakeProfitPercent`), R-multiple (`TakeProfitRMultiple`),
and fixed-price (`TakeProfitFixedPrice`) variants already exist in
`exit.py`.

**Shape (proposed):**

```python
class TakeProfitAtrMultiple(_StrictModel):
    """Exit at +N×ATR(period) above (long) / below (short) entry price.
    Symmetric to StopLossAtrMultiple — exit math identical to the stop
    but in the take-profit direction."""
    kind: Literal["atr_multiple"] = "atr_multiple"
    atr_period: int = Field(ge=2, le=100)
    mult: float = Field(gt=0.0, le=20.0)
```

**Schema file:** `exit.py` gains the class and adds it to the
`TakeProfitMethod` union.

**Note on the type-discriminator conflict:** `TakeProfitMethod` already
has a variant with `kind: Literal["percent"]`, which now matches
`StopLossAtrMultiple`'s `kind: Literal["atr_multiple"]`. The two unions
are SEPARATE (`StopLossMethod` vs `TakeProfitMethod`), so the
discriminator collision is fine — they live under different parents.
But the validator should sanity-check that no spec accidentally puts
a stop-loss variant in a take-profit slot or vice versa (Pydantic does
this for free at the discriminator-union level).

**Iterative engine integration:** `iterative.py` already handles
take-profit exits via `_resolve_take_profit_price()`. Adding a branch
for `TakeProfitAtrMultiple` is symmetric to the existing
`StopLossAtrMultiple` branch:

```python
def _resolve_take_profit_price(method, entry_price, atr):
    if isinstance(method, TakeProfitPercent):
        return entry_price * (1 + method.value)
    if isinstance(method, TakeProfitAtrMultiple):
        return entry_price + method.mult * atr
    if isinstance(method, TakeProfitRMultiple):
        return entry_price + method.r * stop_distance
    if isinstance(method, TakeProfitFixedPrice):
        return method.price
```

**vbt translator integration:** vbt's `from_signals` takes a `tp_stop`
parameter (per-bar fraction). For percent: pass `value` directly. For
ATR: pre-compute the per-bar ATR value, divide by `entry_price` to
get a per-bar fraction, multiply by `mult`. This needs the entry-bar
price snapshot — already available in the translator's `tp_method`
handling for stop_loss_atr_multiple (mirror the existing code path).

**Extraction prompt teaching:** the prompt's "### take_profit" section
(if it has one; check during implementation) gets the new variant
named. Worked example:

```json
{"type": "take_profit",
 "method": {"kind": "atr_multiple", "atr_period": 14, "mult": 2.0}}
```
"Exit at +2 × ATR(14) above entry — a classic R-multiple target where
R is defined by recent volatility rather than a fixed stop distance.
Symmetric to the trailing_atr stop loss, just in the take-profit
direction."

**Drift parity:** the new variant is a pure addition; existing specs
that don't use it are byte-stable. **New gate** (this sub-phase): a
fixture spec using `TakeProfitAtrMultiple` runs through vbt and
iterative paths producing identical trade ledgers.

**Test plan:**

## §5 Sequencing rationale

**Recommended order: v1.2.0 → v1.2.A → v1.2.B → v1.2.C → v1.2.D → v1.2.E → v1.2.F.**

The order is by **isolation** (most isolated first, riskiest last) plus
**reuse** (v1.2.D reuses v1.2.C's implementation pattern):

- **v1.2.A `percentile()`** — purely additive expression wrapper; no
  signature changes; no state. The safest single addition.
- **v1.2.B `bars_since_last`** — schema-additive but requires threading
  `current_bar` into `evaluate_predicate()`. That signature change is
  the highest-risk piece of the batch; doing it second (after the
  no-risk v1.2.A) means the branch is already healthy before the
  change lands.
- **v1.2.C `TimeOfDayCondition`** — new Condition variant, pure addition.
- **v1.2.D `DayOfWeekCondition`** — copy-paste of v1.2.C's pattern with
  a different evaluator. Done immediately after C while the pattern is
  fresh.
- **v1.2.E `TakeProfitAtrMultiple`** — new Exit variant. Symmetric to
  the existing `StopLossAtrMultiple`; the pattern is established but
  exits are more invasive than conditions/expressions (they affect
  trade-close logic in both vbt and iterative paths). Doing it last
  keeps the riskier change isolated.
- **v1.2.F — sign-off** when the suite + drift parity gates + v1
  regression are all green.

## §6 Risk register

| risk | likelihood | severity | mitigation |
|---|---|---|---|
| `evaluate_predicate` signature change breaks call sites | medium | medium | v1.2.B does the threading carefully; the existing tests will catch any miss; pyright will catch the rest |

## §10 Discovered constraints — discovery notes

Two findings worth recording from the design pass:

1. **Scope reduction: 2 of 6 originally-scoped primitives already exist**
   (§3 above). TimeExit (#5) is fully shipped. TakeProfitPercent +
   TakeProfitRMultiple + TakeProfitFixedPrice are fully shipped (only
   TakeProfitAtrMultiple is new). This reduces v1.2's real scope to
   4.5 primitives across 5 implementation sub-phases.

2. **The `evaluate_predicate` signature change (v1.2.B) is the only
   non-purely-additive change in v1.2.** Every other sub-phase is a
   pure addition: new class in a union, new branch in a dispatch,
   nothing existing changes shape. The signature change is bounded
   (3-4 call sites all in the iterative engine), but it's worth
   flagging as the "highest-risk single touch" in this batch. The
   sequencing (v1.2.B second, after the no-risk v1.2.A) keeps the
   branch healthy if the change surfaces unexpected issues.

3. **`_detect_axes` widening** — the overfitting parameter sweep
   (`workers/.../overfitting/parameter_sweep.py:_detect_axes`) currently
   knows about `period` for sma/ema/rsi/wma/adx/keltner and a few other
   fields; it doesn't know about every numeric param in `INDICATOR_RULES`.
   v1.2.A's `PercentileExpr.window` should be added to the axis-detection
   list so the parameter sweep can vary it. This is the same shape as
   the v1.1-todos "generic _detect_axes widening" entry — could be
   bundled with v1.2.A's prompt teaching commit, or deferred to a
   separate housekeeping pass.

**No primitive turned out to have higher complexity than estimated.**
None require architectural change beyond schema additions. The
design-pass STOP threshold did not fire.
