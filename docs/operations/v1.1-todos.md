## `build_overfitting_metrics_payload` reads wrong key names

**Where:** `scripts/trader_seed_strategy.py`, `build_overfitting_metrics_payload()`.

**Symptom:** The seeded `trader_strategy_versions.overfitting_metrics.deflated_sharpe` subtree always reads as `{"deflated_sharpe": null, "probabilistic_sharpe": null}`, regardless of what the source `overfitting_analyses.deflated_sharpe_json` actually contains.

**Root cause:** The script reads keys `deflated_sharpe` and `probabilistic_sharpe` from `deflated_sharpe_json`. The actual JSON shape in the DB (from `marketmind_shared.schemas.overfitting.DeflatedSharpeResult`) uses `deflated_sharpe_ratio` and `probability_strategy_is_real` — different key names. `.get()` returns `None` silently.

**Functional impact:** None. The trader does **not** read `overfitting_metrics` — the column is purely for operator inspection via `GET /trader/strategies`. The admin `approve_paper` validator only checks `backtest_metrics`, which is populated correctly.

**Cosmetic impact:** When an operator views the seeded version's `overfitting_metrics`, the `deflated_sharpe` subtree shows `null` instead of the real DSR. To see the real DSR they have to follow `marketmind_spec_id` → `overfitting_analyses.deflated_sharpe_json`.

**Fix (v1.1):**
```python
# In build_overfitting_metrics_payload:
"deflated_sharpe": {
    "deflated_sharpe_ratio": deflated_sharpe_json.get("deflated_sharpe_ratio"),
    "probability_strategy_is_real": deflated_sharpe_json.get("probability_strategy_is_real"),
    "observed_sharpe": deflated_sharpe_json.get("observed_sharpe"),
},
```
…and either rename the existing seeded rows' subtrees via a one-shot UPDATE, or accept the historical rows keep the null shape and only new seeds get the correct values.

**Discovered during:** 2026-05-19 trader seed of `d64f24bc` + `8045af6b`.

## `/data` bind mount not writable by the worker container

**Where:** `docker-compose.yml` (`worker` + `api` mount `./data:/data`); `infra/worker.Dockerfile` (runtime `USER`).

**Symptom:** First `ingest_youtube` job on a fresh checkout fails with `PermissionError: [Errno 13] Permission denied: '/data/cache'` when the worker tries to create its on-disk cache dir.

**Root cause:** The bind-mounted host `./data` is owned by the host user (uid 1000, mode 755). The worker image runs as `uid 999 (app)`, which gets only read+execute on the mount and cannot create `/data/cache`. Invisible on macOS (Docker Desktop remaps mount ownership); surfaces on a Linux/WSL2 host where bind-mount uids pass through verbatim.

**Functional impact:** Blocks every worker job that writes under `/data` (YouTube audio cache, market-data parquet) until the host dir is made writable.

**Current mitigation:** `chmod 777 ./data` on the Windows/WSL2 host — works, but is a manual per-machine step and leaves the dir world-writable.

**Fix (v1.1):** Align the uids — a Dockerfile `USER` directive matching the typical host uid, or a compose `user:` directive on `worker`/`api`. Also relevant to the Phase 5.2b Railway deployment, where the volume mount carries the same ownership question.

**Discovered during:** 2026-05-20 Mac->Windows migration + trader re-seed.

## Trader candle backfill depth vs template warmup

**Where:** trader market-data ingestion (`workers/trader/ingestion.py`); `trader_candles` table.

**Symptom:** On a fresh DB the runner loads its strategy versions (`versions_loaded=2`) but logs `signal_pair_insufficient_history have_bars=199 need_bars=205` and emits no signals until enough new 4h candles close (~24h).

**Root cause:** The trader backfills ~200 bars of history; the seeded `ma_trend` / `breakout` templates use `trend_ema_period=200`, so `min_bars_needed()` is `200 + 5 = 205`. The backfill lands ~6 bars short of the warmup requirement.

**Functional impact:** None lasting — the runner gains one bar per 4h period and self-heals into signal evaluation within ~24h of a fresh-DB start. Only a fresh DB is affected; an established DB already holds the history.

**Decision:** Accepted as-is for v1 — the ~24h fresh-DB warmup is tolerable. Revisit only if a future strategy/template needs a longer warmup that makes the gap material (then deepen the backfill, e.g. target `max(min_bars_needed) + margin`).

**Discovered during:** 2026-05-20 Mac->Windows migration + trader re-seed.

## Article ingestion can't handle JS-rendered or bot-walled sites

**Where:** `workers/src/marketmind_workers/services/ingest.py`, `ingest_article()` — static-HTML fetch + trafilatura extraction.

**Symptom:** Article URLs on modern sites fail ingestion in one of two shapes — `ContentTooShortError` (the fetch returns a near-empty pre-JavaScript shell; trafilatura extracts a handful of chars) or `NetworkError: no HTML returned` (the fetcher gets a non-200 / challenge response).

**Root cause:** `ingest_article` fetches *static* HTML and extracts text from it. Two increasingly common patterns defeat that: (a) the article body is rendered client-side by JavaScript, so the served HTML is just an app shell; (b) the site gates content behind an anti-bot wall ("Verifying you are not a robot…"). A static fetcher sees only the shell or the challenge page, never the article.

**Evidence:** the 2026-05-20 article-gauntlet run. quantifiedstrategies.com served a bot-verification wall identically to trafilatura, `curl` (browser UA), WebFetch, *and* the Internet Archive's crawler — every automated fetcher. Only a real browser session renders the article.

**Functional impact:** any article on a JS-rendered or bot-walled host is un-ingestable; plain static-HTML blogs still work. The usable corpus for article ingestion narrows as more sites move to client-side rendering and Cloudflare-style bot walls.

**Fix (v1.1):** add a headless-browser fetcher (e.g. Playwright + Chromium) that loads the page, runs its JavaScript, and hands the rendered DOM to trafilatura. Trigger it as a second pass when the static fetch returns too little text (reuse the `ContentTooShortError` threshold as the signal). It is a heavy dependency and slower per fetch, so keep it opt-in / fallback-only, not the default path. It won't defeat every CAPTCHA, but it handles the common JS-render and soft-challenge cases.

**Discovered during:** 2026-05-20 article-gauntlet evaluation (5 URLs — 2 JS/bot-walled, 3 unrelated dead links).

## Phase A capability boundary — when stateful primitives are load-bearing

Surfaced 2026-05-21 during first v2 stateful seeding attempts (two extractions, both came back Tier-1). v2's `regime_state` primitive is correctly chosen by the LLM only when:

1. The regime acts as a filter on a separate entry signal (read as state on non-transition bars), AND
2. The regime has hysteresis — distinct enter and exit thresholds — so the latched value cannot be recomputed instantaneously.

Same-threshold trend-followers ("long when close > EMA-200, exit when close < EMA-200") are degenerate stateful — they reduce exactly to a crossover pair and the LLM correctly extracts them as Tier-1.

This is not a bug. It's a design boundary worth registering: not every "stateful-feeling" strategy exercises Phase A's stateful machinery. Strategies that do:

- Hysteresis-banded regime filters (e.g. enter when bullish-and-pullback, exit when bearish OR signal-fades, with hysteresis on the regime threshold).
- Indicators that are themselves stateful (Supertrend's recursive direction — pending indicator expansion).
- Outcome-dependent rules (Turtle's skip-after-winner — exercised via `prior_signal`).
- Ratchet variables (trailing stops that only move favorably).

Implication for strategy discovery: when looking for "real-world strategies that exercise Phase A," prefer strategies that explicitly mention regime filters with bands, hysteresis, trailing stops, or skip-after-X rules. Pure trend-followers are Tier-1 and validate the SpecTemplate pipeline but not the stateful machinery.

## Extraction prompt teaching audits

Two consecutive sessions (2026-05-21 regime_state, 2026-05-22 highest/lowest) found the same bug shape: prompt under-teaches a load-bearing primitive, real strategy hits the gap, extraction degrades to refusal. The LLM is generally capable of using v2 primitives correctly when shown a load-bearing example, but a degenerate example (regime_state) or no example at all (highest/lowest params.source) silently teaches the wrong thing.

Bias toward proactive auditing: when a new primitive ships, the prompt's example for it must demonstrate the LOAD-BEARING use case, not the simplest possible reduction. Review each primitive's example against the test: "if the LLM only ever saw this example, would it know how to use the primitive in a non-trivial case?"

This is also a discovery-flow finding: real-world strategies hitting these gaps is how we find them. Each new candidate strategy that hits a fresh teaching gap should produce both the immediate prompt fix AND an audit pass of other primitives.

## "spot" exchange string in extracted specs (Phase B seedability runs)

Both B.7 (EMA crossover 1H) and B.9 (BB+EMA200 15m) produced specs
with `instrument.exchange = "spot"` instead of `"binance"`. The
FeeModel / SlippageModel `_exchange_key()` mapping only translates
`"binance"` → `"binance_spot"`; `"spot"` falls through to the
conservative 10 / 5 bps fallback path. That fallback happens to match
the binance_spot table entry exactly for BTC/USDT, so backtest math
was unaffected in both runs.

**Risk:** if a future extraction outputs another non-mappable exchange
string (e.g. `"crypto"`, `"futures"`), and the table later carries
non-default fees for that venue, the fallback path will silently use
the wrong (conservative-default) numbers.

**Fix (v1.1):** add an explicit "exchange" question to the extraction
prompt template (e.g. "if the source doesn't specify an exchange,
default to `'binance'`"), and either:

- map a small set of common venue-type-as-exchange strings (`"spot"`,
  `"crypto"`) to `"binance_spot"` defensively in `_exchange_key()`, or
- promote unknown-exchange to a strict validator error so the
  extraction job surfaces the issue rather than silently using
  fallback fees.

**Discovered during:** 2026-05-23 B.7 + B.9 seedability runs.

## `max_drawdown > 80 %` as a separate gauntlet refusal flag

B.9's BB+EMA200 15m strategy posted a 99.6 % max drawdown across
its 6-year backtest. The composite gauntlet correctly classified it
`likely_overfit` (score 61/100), but the composite's per-bucket
scores don't distinguish "lost essentially all capital" from
"sharpe-of-noise overfit" — both produce similar composite scores
via different mechanisms.

**Idea:** add a hard-refusal short-circuit ahead of the composite —
if `metrics.max_drawdown_pct > 0.80`, mark the gauntlet result as
`likely_overfit` regardless of composite, with a dedicated
explanation pointing at the drawdown number specifically. Saves the
operator from having to read the per-bucket scores to understand
"this strategy actively destroys capital, not just doesn't beat
benchmark."

**Trade-off:** any hard short-circuit risks edge-case false-positives
(a recovering strategy that briefly drew down 81 % is meaningfully
different from one that bled to 99 %). Probably want a softer
two-tier signal: 80-95 % adds a "capital_destruction" warning to the
composite explanation but doesn't change the verdict; >95 % triggers
the hard refusal.

**Discovered during:** 2026-05-23 Phase B.9.
