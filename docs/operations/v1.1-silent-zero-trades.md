# Silent zero-trade backtests from logically-degenerate specs (v1.1)

## TL;DR

Three extractions in the DB had specs that were schema-valid but
logically degenerate (e.g. `close crosses above ATR(10)` — BTC price
is always ≫ ATR, the crossover never fires). Their backtests ran
cleanly and produced 0 trades. The UI showed "0 trades" as if it were
a legitimate "strategy too restrictive" result. **There was no way to
distinguish a real no-signals strategy from a silently mis-extracted
spec.** Added a `SignalDiagnostics` classifier and persisted it on
`BacktestRun.entry_diagnostics` so the result page can warn.

## Investigation

The user reported two cases:

- `0b700544` Turtle Trading — flagged validation errors but a backtest
  "still ran" with 0 trades.
- `4307e4f3` Supertrend — extraction completed, backtest produced
  0 trades.

**Correction of the first claim:** `0b700544` correctly **downgraded to
`not_extractable` with `spec=None`** when the schema validator
rejected its spec. No backtest exists for it. The downgrade
mechanism in `extract.py:_downgrade_to_refusal` is working as
intended; validation-error specs never reach the engine.

`4307e4f3` is actually a `backtest_id`, not an extraction. It belongs
to extraction `d31ec081` (Bitcoin Supertrend Strategy). That spec
extracted with `verdict=partially_extractable`, schema-validated, and
ran a backtest that produced 0 trades.

Three such cases live in the DB:

| short_id | name | tf | bars | why 0 trades |
|---|---|---|---|---|
| `d31ec081` | Bitcoin Supertrend Strategy (ATR=10) | 1d | 2331 | Entry `close crosses above ATR(10)`. BTC close ~60k, ATR ~1k. Crossover requires a transition, which never happens after the first bar where price > ATR (everywhere from 2020 onward). |
| `f54f8542` | SuperTrend BTCUSD 4H | 4h | 13985 | Same Supertrend mis-extraction — LLM dropped the price-level part of `(HL2) ± mult×ATR` and used raw ATR as threshold. |
| `534ede07` | RSI EMA Crossover Trend | 1h | 55888 | Entry includes `EMA(9) of close crosses above constant 65`. EMA(9) of BTC close is always ~60k, never near 65 — crossover transition impossible. |

All three: extraction LLM produced something the schema accepts but
that doesn't compute meaningful signals on the asset. The backtest
engine isn't buggy; it's doing exactly what the spec says.

### Where the silence came from

`workers/backtest/translator.py:_eval_compare` / `_eval_crossover`
return pandas boolean Series. pandas semantics: `NaN > x` evaluates
to `False`. After eval, `build_signals` did:

```python
entries=entries.fillna(value=False).astype(bool)
```

So both:
- Deterministic `False` (indicators warmed up, comparison evaluated
  to False) and
- NaN (one side never produced a value)

collapsed to `False` before reaching vbt. **No counter, no log, no
exception.** The engine could not distinguish "real no-signals
strategy" from "all-NaN never-evaluable expression."

## The fix

Added `SignalDiagnostics` to `shared/schemas/backtest_run.py`, computed
in `translator.build_signals` BEFORE the `.fillna` step, persisted on
`BacktestRun.entry_diagnostics` (new optional field, backward-compatible
with historical rows that store `None`).

The classifier buckets each entry-Series bar into:
- **nan_warmup_count** — bars 0 to longest-indicator-period that are NaN (expected)
- **nan_post_warmup_count** — bars after warmup that are still NaN (surprising)
- **true_count** — bars where the condition fired
- **deterministic_false_count** — bars where the condition returned a clean False

Then classifies into:
- `NONE` — at least one True signal fired
- `CONDITIONS_NEVER_MET` — clean Falses everywhere, indicators warmed up. Strategy is either legitimately too restrictive OR (more commonly) logically degenerate. The three cases above all land here.
- `EVALUATION_DEGRADED` — > 50% of post-warmup bars NaN. Almost always a bug: missing indicator source, mismatched expression types, window too short for the indicators.

## End-to-end verification

Re-ran `d31ec081` (Bitcoin Supertrend) with the new diagnostics. The
persisted `BacktestRun.entry_diagnostics` JSONB:

```json
{
    "failure_mode": "conditions_never_met",
    "bars_evaluated": 2331,
    "nan_warmup_count": 0,
    "nan_post_warmup_count": 0,
    "true_count": 0,
    "deterministic_false_count": 2331,
    "warmup_bars": 11
}
```

Reads: "the condition was evaluated on every one of 2331 bars,
warmed up cleanly, and returned a deterministic False on every
single one." That's exactly the right diagnosis — the engine isn't
broken; the SPEC is degenerate. Plus a structlog event
`signal_diagnostics_computed` carrying the same fields.

## What's NOT included (the user explicitly deferred)

**Layer 3 — UI surfacing.** The diagnostic data is now persisted on
every new `BacktestRun`, but `api/routes/backtests.py` doesn't surface
`entry_diagnostics` in its response model and the result-page UI doesn't
render a warning when `failure_mode != NONE`. That's a separate
ticket. The data is there to read whenever the UI is ready.

Historical backtest rows have `entry_diagnostics: None` (pre-fix
rows weren't computed). The result page would need to handle that as
"diagnostics unavailable for this run" rather than treating it as a
success signal.

## What "future you" should do if this rhymes

When a user reports a strategy "got 0 trades, is the engine broken?":

1. Pull `backtest_results.result_json -> run -> entry_diagnostics` for
   the row. If null, it's a pre-fix backtest; ask them to re-run.
2. Read the `failure_mode` field:
   - `none` — at least one entry fired. 0 trades comes from exit
     gates or risk filters, not from the entry condition.
   - `conditions_never_met` — entry condition deterministic-False
     throughout. Almost certainly a degenerate spec (compare across
     wildly different scales, crossover that requires an impossible
     transition, threshold value that's never reached).
   - `evaluation_degraded` — > 50% post-warmup NaN. Suspect missing
     indicator source, mismatched expression types, or a window
     shorter than the longest indicator period.
3. Read the `signal_diagnostics_computed` structlog event for the
   same backtest in the worker logs — has the same fields.

## Hard-won knowledge worth adding to CLAUDE.md

> **`fillna(False)` on a boolean signal Series silently collapses
> three distinct states into one.** Deterministic True, deterministic
> False, and NaN (couldn't evaluate) all become Python bool by the
> time they hit vbt. If you need to distinguish them downstream
> (for diagnostics, debugging, or operator-facing warnings), capture
> the counts BEFORE the fillna step. Once the cast happens, the
> information is gone.

## Relationship to prior incidents

This is the third v1.1-class downstream-bottleneck incident in this
session:

1. `MAX_OUTPUT_TOKENS = 4096` truncated tool_use payloads silently —
   "tool_use payload missing or malformed `report` field" was the
   only signal. Fixed with diagnostic logging + raising to 16384.
2. `MAX_WALL_CLOCK_SECONDS = 60` was too tight for the new token
   ceiling. Fixed with `generation_seconds` logging + raising to 240.
3. This one: `.fillna(False)` silently collapsed three states. Fixed
   with `SignalDiagnostics` capture before the cast.

Common shape: **a layer in the pipeline is making a coercion or
budget decision that destroys information, and the only signal that
information loss happened is a downstream symptom.** Adding
diagnostic capture at the coercion site — before the information is
lost — turns the symptom into a clean explanation.
